[role="lead"]
Esta solução fornece uma visão geral e detalhes para a implantação do Oracle no Google Cloud NetApp Volumes como storage de banco de dados primário sobre o protocolo NFS e o banco de dados Oracle é implantado como um banco de dados de contêiner independente com dNFS habilitado.



== Propósito

Executar cargas de trabalho Oracle exigentes em desempenho e sensíveis à latência na nuvem pode ser um desafio. Google Cloud NetApp Volumes (GCNV) facilita para profissionais de linha de negócios (LOB) e de storage de empresas migrarem e executarem cargas de trabalho Oracle exigentes sem alteração de código. Google Cloud NetApp Volumes é amplamente utilizado como o serviço subjacente de armazenamento de arquivos compartilhado em vários cenários, como novas implantações ou migração (lift and shift) de bancos de dados Oracle de ambientes locais para o Google Cloud.

Esta documentação demonstra a configuração dos volumes GCNV DB no console do Google Cloud e a implantação simplificada de bancos de dados Oracle no Google Cloud NetApp Volumes via pontos de montagem NFS usando automação Ansible. O banco de dados Oracle é implantado em uma configuração de container database (CDB) e pluggable databases (PDB) com o protocolo Oracle dNFS habilitado para aumentar o desempenho. Além disso, demonstra uma estratégia detalhada de backup, restauração e clonagem de banco de dados e fornece um conjunto de ferramentas de automação para o gerenciamento de backup de banco de dados Oracle no Google Cloud.

Esta solução aborda os seguintes casos de uso:

* Implantação automatizada de banco de dados Oracle no Google Cloud NetApp Volumes no Google Cloud com protocolo NFS.
* Backup, recuperação e clonagem de banco de dados Oracle no Google Cloud NetApp Volumes no Google Cloud.




== Público

Esta solução é destinada às seguintes pessoas:

* Um DBA que gostaria de implantar o Oracle no Google Cloud NetApp Volumes.
* Um arquiteto de soluções de banco de dados que gostaria de testar cargas de trabalho do Oracle no Google Cloud NetApp Volumes.
* Um administrador de storage que gostaria de implantar e gerenciar um banco de dados Oracle no Google Cloud NetApp Volumes.
* Um proprietário de aplicativo que gostaria de configurar um banco de dados Oracle no Google Cloud NetApp Volumes.

=== Fatores-chave para consideração de implantação

* *Nível de serviço do pool de storage GCNV e taxa de transferência.* GCNV oferece quatro níveis de serviço diferentes: Standard, Premium, Extreme e Flex. Para os níveis de serviço Standard, Premium e Extreme, a taxa de transferência de E/S é determinada e fixada com base no tamanho de um volume de dados. A taxa de transferência total de E/S é limitada com base no dimensionamento do pool de storage. Para o nível de serviço Flex, a taxa de transferência de E/S não é fixa em relação ao tamanho de um volume de dados, mas compartilhada entre todos os volumes de dados e limitada ao nível de dimensionamento do pool de storage. Isso acomoda bancos de dados com volumes menores e picos ocasionais de IOPS. Para referência, os níveis de serviço Standard, Premium e Extreme fornecem taxas de transferência de 16 KiB/s por GiB, 64 KiB/s por GiB e 128 KiB/s por GiB, respectivamente. Por outro lado, o nível de serviço Flex oferece até 5 GiB/s por pool de storage com configuração de desempenho personalizada. É importante dimensionar corretamente o nível de serviço e o pool de storage com base na taxa de transferência de E/S e nos requisitos de IOPS esperados da carga de trabalho do banco de dados Oracle.
* *Layout de storage de banco de dados.* Nesta implantação automatizada do Oracle, provisionamos três volumes de banco de dados para cada banco de dados para hospedar binário, dados e logs do Oracle por padrão. Os volumes são montados no servidor Oracle DB como /u01 - binário, /u02 - dados, /u03 - logs via NFS. Arquivos de controle duplos são configurados nos pontos de montagem /u02 e /u03 para redundância.
* *Configuração do dNFS.* Ao usar o Oracle dNFS (disponível desde o Oracle 11g), um banco de dados Oracle executado no Google Compute Engine com armazenamento Google Cloud NetApp Volumes pode gerar significativamente mais E/S do que o cliente NFS nativo. A implantação automatizada do Oracle configura o dNFS no NFSv3 por padrão.
* *Snapshot e Vault backup.* Em vez do backup tradicional de banco de dados RMAN, a NetApp recomenda implementar snapshot e vault backup eficientes em storage e consistentes com o aplicativo para obter backup de snapshot rápido (segundos), restauração rápida (minutos), recuperação e clonagem do banco de dados a partir de snapshots ou backups no storage vault. Um snapshot é uma cópia pontual dos volumes de dados do banco de dados que pode ser criada em segundos e não consome espaço de storage adicional no momento da criação. Ele coexiste com os volumes primários do banco de dados e pode ser perdido se os volumes primários forem comprometidos. Um vault backup é uma cópia do snapshot que é armazenada no storage de objetos e em um local diferente para fins de recuperação de desastres.
* *Considerações sobre RTO/RPO.* Ao configurar estratégias de backup de banco de dados, é importante considerar os requisitos de Recovery Time Objective (RTO) e Recovery Point Objective (RPO). Embora o impacto no desempenho do banco de dados seja mínimo com backup baseado em Snapshot, existe uma relação de compromisso entre a frequência dos backups, que afeta o RTO/RPO, e o gasto com espaço de storage. Backups mais frequentes podem ajudar a alcançar um RTO/RPO menor, mas podem aumentar os custos de espaço de storage. É importante encontrar o equilíbrio certo com base nos requisitos de negócios e no orçamento. A solução de automação fornece um conjunto de ferramentas de automação baseado em Ansible playbook para gerenciar backup de banco de dados Oracle com política de retenção e agendamento de backup configuráveis pelo usuário.




== Implantação da solução

As seções a seguir fornecem procedimentos passo a passo para a implantação automatizada do Oracle 21c e backup, recuperação e clonagem de banco de dados no Google Cloud NetApp Volumes com volumes de banco de dados montados diretamente via NFS em VMs do Google Cloud Compute Engine como servidores de banco de dados.



=== Pré-requisitos para implantação

[%collapsible]
====
A implantação requer os seguintes pré-requisitos.

. Uma conta do Google Cloud foi configurada e as configurações necessárias de VPC e rede foram criadas no projeto específico da sua conta do Google Cloud.
. A partir do console do Google Cloud, implante VMs do Google Compute Engine como servidores de banco de dados Oracle. Certifique-se de desativar o firewall e habilitar a autenticação SSH de chave privada/pública da VM com um usuário administrador para os servidores de banco de dados para automação. Consulte o diagrama de arquitetura na seção anterior para detalhes sobre a configuração do ambiente.
+

NOTE: Certifique-se de ter alocado pelo menos 50G no disco raiz da máquina virtual para ter espaço suficiente para armazenar os arquivos de instalação do Oracle e adicionar o arquivo de swap do sistema operacional.

. Provisione uma máquina virtual Linux como o nó controlador do Ansible com a versão do Ansible e do Git instalada, conforme listado no README do kit de ferramentas de automação. Para obter ajuda com a automação do Ansible, consulte o seguinte link: link:https://docs.netapp.com/us-en/netapp-solutions-dataops/automation/getting-started.html["Introdução à automação de soluções da NetApp ^"^] na seção -
`Setup the Ansible Control Node for CLI deployments on RHEL / CentOS` ou 
`Setup the Ansible Control Node for CLI deployments on Ubuntu / Debian`.
+

NOTE: O nó controlador do Ansible pode estar localizado tanto em infraestruturas locais quanto no Google Cloud, desde que consiga alcançar as máquinas virtuais do servidor de banco de dados do Google Cloud via porta ssh.

. Clone uma cópia do kit de ferramentas de automação de implantação do NetApp Oracle para NFS.
+
[source, cli]
----
git clone https://bitbucket.ngage.netapp.com/scm/ns-bb/na_oracle_deploy_nfs.git
----

=== Implantação automatizada do Oracle com playbook do Ansible



==== Arquivos de parâmetros de automação

[%collapsible]
====
O playbook do Ansible executa tarefas de instalação e configuração de banco de dados com parâmetros predefinidos. O toolkit atualmente suporta a implantação do banco de dados Oracle versão 19c e 21c. Para esta solução de automação do Oracle, existem três arquivos de parâmetros definidos pelo usuário que precisam de entrada do usuário antes da execução do playbook.

* hosts - definem alvos nos quais o manual de automação está sendo executado.
* vars/vars.yml - o arquivo de variável global que define variáveis que se aplicam a todos os destinos.
* host_vars/host_name.yml - o arquivo de variáveis locais que define variáveis que se aplicam somente a um destino nomeado.  No nosso caso de uso, esses são os servidores Oracle DB.


Além desses arquivos de variáveis definidos pelo usuário, há vários arquivos de variáveis padrão que contêm parâmetros padrão que não exigem alterações, a menos que seja necessário.  As seções a seguir mostram como configurar os arquivos de variáveis definidas pelo usuário.

==== Execução do manual de instruções

[%collapsible]
====
Há um total de cinco manuais no kit de ferramentas de automação.  Cada um executa blocos de tarefas diferentes e atende a propósitos diferentes.

....
0-all_playbook.yml - execute playbooks from 1-4 in one playbook run.
1-ansible_requirements.yml - set up Ansible controller with required libs and collections.
2-linux_config.yml - execute Linux kernel configuration on Oracle DB servers.
4-oracle_config.yml - install and configure Oracle on DB servers and create a container database.
5-destroy.yml - optional to undo the environment to dismantle all.
....
Há três opções para executar os playbooks com os seguintes comandos.

. Execute todos os manuais de implantação em uma execução combinada.
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml
----
. Execute os playbooks um de cada vez com a sequência numérica de 1 a 4.
+
[source, cli]]
----
ansible-playbook -i hosts 1-ansible_requirements.yml -u admin -e @vars/vars.yml
----
+
[source, cli]
----
ansible-playbook -i hosts 2-linux_config.yml -u admin -e @vars/vars.yml
----
+
[source, cli]
----
ansible-playbook -i hosts 4-oracle_config.yml -u admin -e @vars/vars.yml
----
. Execute 0-all_playbook.yml com uma tag.
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t ansible_requirements
----
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t linux_config
----
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t oracle_config
----
. Desfazer o ambiente
+
[source, cli]
----
ansible-playbook -i hosts 5-destroy.yml -u admin -e @vars/vars.yml
----

=== Backup do banco de dados Oracle com o Google Cloud NetApp Volumes



==== Snapshot e backup do banco de dados Oracle

[%collapsible]
====
Para facilitar a configuração fácil do backup do banco de dados Oracle, a equipe de Engenharia de Soluções da NetApp desenvolveu um playbook Ansible para automatizar o backup do banco de dados Oracle com política de retenção e agendamento de backup configuráveis pelo usuário. O playbook utiliza os recursos de snapshot e backup em cofre do Google Cloud NetApp Volumes para realizar backup de snapshot rápido (em segundos), restauração rápida (em minutos), recuperação e clonagem do banco de dados a partir de snapshots ou backups no cofre de storage.

. Clone uma cópia do kit de ferramentas de automação de backup do banco de dados Oracle NetApp para GCNV.
+
[source, cli]
----
git clone https://bitbucket.ngage.netapp.com/scm/ns-bb/na_oracle_bkup_gcnv.git
----
+

NOTE: No momento, o kit de ferramentas só pode ser acessado por usuários internos da NetApp com acesso ao bitbucket.  Para usuários externos interessados, solicite acesso à sua equipe de conta ou entre em contato com a equipe de engenharia de soluções da NetApp .

. Leia o arquivo README no kit de ferramentas e siga as instruções abaixo sobre como configurar e agendar o trabalho de backup a partir do crontab ou de qualquer outra ferramenta de agendamento. O playbook foi projetado para ser executado em um nó controlador Ansible com acesso às VMs do servidor de banco de dados Oracle e ao Google NetApp Volumes. Ele criará Snapshots consistentes com o aplicativo dos volumes do banco de dados com base no agendamento definido e na política de retenção, e copiará os Snapshots para o vault para fins de recuperação de desastres.
. Por padrão, o playbook cria um backup de Snapshot diário e Snapshots horários a cada hora. A retenção padrão de Snapshots é de 7 Snapshots diários e 24 Snapshots horários. Quaisquer Snapshots adicionais que excedam a retenção são removidos, mantendo cópias rotativas de 7 Snapshots diários e 24 Snapshots horários. Você pode ajustar a frequência de backup e a retenção com base nos seus requisitos de RTO/RPO e na consideração do custo de storage. Enquanto o backup de Snapshot diário inclui todos os volumes de banco de dados, o backup de Snapshot horário inclui apenas os volumes de log e economiza espaço de storage. Durante o backup de Snapshot diário, o playbook também remove arquivos de log de arquivamento do Oracle com base na retenção definida para economizar espaço de storage nos volumes de log do banco de dados.
. O seguinte é um exemplo de entrada no crontab para criar backup de Snapshot e copiar para o cofre.
+
....

[admin@ansiblectl na_oracle_bkup_gcnv]$ crontab -l
0 0 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_snapshot_daily.sh
0 1-23 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_snapshot_hourly.sh
5 0 * * 7 /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_weekly.sh
5 0 * * 1-6 /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_daily.sh
5 1-23 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_hourly.sh
....
. O backup do vault pode ser executado em intervalos semanais, diários ou horários, dependendo dos seus requisitos de RTO/RPO. Enquanto o backup semanal e diário inclui todos os volumes de banco de dados, o backup do vault por hora inclui apenas os volumes de log para economizar espaço de storage. O primeiro backup do vault leva mais tempo, pois cria a linha de base. Uma vez estabelecido o backup de linha de base, todos os backups adicionais do vault são incrementais, utilizando a metodologia incremental forever. Todos os backups do vault são criados a partir dos snapshots mais recentes e consistentes com o aplicativo no momento da execução, para garantir a recuperabilidade. Ao contrário dos backups típicos de linha de base e incrementais, os dados do backup de linha de base do vault são consolidados em cada backup incremental. Em outras palavras, cada backup incremental do vault contém o conjunto de dados completo e pode ser usado para recuperação sem a necessidade de restaurar o backup de linha de base. Essa abordagem simplifica o gerenciamento de backup e o processo de recuperação, ao mesmo tempo que proporciona uma utilização eficiente do storage no vault. Não há necessidade de se preocupar com a cadeia de backup e dependências ao excluir qualquer backup, já que todos os backups são independentes nessa abordagem. O script de automação de backup ajusta automaticamente os backups para atender à meta de retenção definida.

==== Recuperação de banco de dados Oracle para um novo host usando backups do cofre do Google Cloud NetApp Volumes

[%collapsible]
====
Em caso de falha que exija recuperação para um novo host, como quando o host original não está mais disponível e os volumes primários do banco de dados estão inacessíveis, você pode usar os backups do vault do Google Cloud NetApp Volumes para restaurar o banco de dados Oracle no novo host. O processo é semelhante à recuperação in-place usando Snapshots, mas, em vez de reverter para um Snapshot, você restaurará o banco de dados a partir de um backup do vault. Isso permite recuperar o banco de dados em um host diferente, o que pode ser útil em cenários em que o host original não está mais disponível ou sofreu uma falha de hardware. As etapas para restaurar a partir de um backup do vault são as seguintes:

. Identifique o backup do vault que contém os dados que você deseja recuperar. Você pode usar o Google Cloud Console ou a ferramenta de linha de comando gcloud para listar os backups do vault disponíveis para o seu volume de banco de dados Oracle. Clique nos três pontos no final da lista de backups do vault e em  `Show more` para visualizar as opções. Selecione a  `Create new volume from backup` para restaurar a partir do backup do vault selecionado. Repita para todos os volumes de banco de dados. ==== Clonagem de banco de dados Oracle para um novo host usando snapshots do Google Cloud NetApp Volumes ou backups de vault

[%collapsible]
====
A clonagem do banco de dados Oracle para um novo host usando snapshots ou backups de vault do Google Cloud NetApp Volumes é idêntica à seção anterior que descreve como usar snapshots ou backups de vault do Google Cloud NetApp Volumes para restaurar e recuperar um banco de dados Oracle em um novo host em caso de falha. Mas renomear o banco de dados clonado pode ser uma etapa adicional desejada, que pode ser facilmente feita usando o utilitário Oracle dbnewid. O clone do banco de dados pode ser útil para testes de UAT, desenvolvimento ou outros propósitos.
